# Select Paper

帮我围绕一个研究主题，或一篇给定的种子论文，快速筛选同主题下质量相对高的论文。默认输出 5 篇。

主要目标：

- 支持两种输入：
  - 一个主题、问题、任务名或方向关键词
  - 一篇种子论文的标题、链接、arXiv id 或其他可唯一定位的信息
- 联网搜索同主题论文，并优先寻找已经发表、质量较高、仍值得阅读的代表性工作。
- 默认并行启动 subagent，加速“候选发现、外部评估、快速内部评估、结果汇总”。
- 最终给我的结果以“论文标题 + 一段话介绍内容”为主，不要输出又长又散的检索过程。

评估原则：

1. 外部评估信号优先
   - 如果论文已正式发表，优先根据其会议或期刊 venue 进行质量判断。
   - 如果论文尚未正式发表、主要停留在 arXiv，则重点评估：
     - 主要作者的 Google Scholar 学术影响力
     - 作者所属实验室、团队或机构在该方向的公认声誉
   - 除非论文明显属于纯理论工作或综述 / survey / benchmark / taxonomy 类型，否则必须有可信的开源代码。
   - 对有代码的论文，还要结合以下信息综合判断：
     - 发布时间是否较新
     - GitHub star 数
     - 最近活跃情况是否正常
     - 是否看起来是作者官方实现或高可信实现

2. 内部评估信号只做快速判断
   - 不做超长精读，但也不能只看摘要就下结论。
   - 在尝试总结论文内容时，快速扫一遍全文，并围绕少数核心指标做快速评估。
   - 内部快速信号重点看：
     - 论文要解决的问题是否清楚且值得做
     - 核心方法或观点是否明确，不是只靠术语堆砌
     - 实验部分是否完整到足以支持主要结论
     - 是否包含强 baseline、关键对比、必要消融或误差分析中的至少一部分
     - 是否存在明显的“结论大于证据”的风险
   - 如果内部信号不强，但外部信号很强，可以保留为候选，但要降低排序并明确说明保留理由。

执行策略：

1. 先识别输入类型
   - 如果我给的是主题，先明确主题边界，并生成检索关键词、同义词、相关任务名、常见 venue 线索。
   - 如果我给的是种子论文，先确认唯一论文身份，再围绕以下线索扩展同主题候选：
     - Semantic Scholar / Google Scholar 的相似论文
     - 相关工作、共引关系、后续引用关系
     - 同任务、同 benchmark、同方法流派的论文
   - 如果种子论文无法唯一定位，先给我最多 3 个候选，不要直接猜。

2. 默认动态并行启动多个 subagent，不设固定上限
   - subagent 数量应根据候选论文规模、信息缺口和需要核验的维度动态扩展，不要被固定成少数几个角色。
   - 可以同时搜索、核验、评估多篇论文；不要一篇一篇串行处理，优先按“论文批次”或“评估维度”拆分成并行任务。
   - 可以按职责分工，也可以按论文分片：
   - `candidate-agent`
     - 扩展候选论文池
     - 目标是快速找到 10-20 篇可能相关的候选，而不是只找 5 篇
   - `venue-agent`
     - 可以负责一批论文的发表状态、venue 层级、作者与实验室声誉核验
   - `code-agent`
     - 可以负责一批论文的开源代码、项目主页、Papers with Code、作者仓库与 star 信息核验
   - `content-agent`
     - 可以对一批优先候选做快速全文扫描
     - 提取论文要解决的问题、核心方法、实验支撑强弱与明显风险点
   - 如果候选较多，可以继续派生更多同类 subagent 并行处理，不要因为已有 agent 名称而停止扩展。
   - 主 agent 负责持续汇总各 subagent 的结果，去重、排序与最终选择。

3. subagent 的返回要求
   - 只返回和筛选直接相关的结论，不要写成长篇综述。
   - 每条候选尽量区分：
     - 已确认事实
     - 基于证据的推断
     - 未确认项
   - 对关键判断尽量附来源名称或链接。
   - 如果没有找到可靠证据，要尽快写“未找到可靠证据”，不要无限扩展搜索。

工作流：

1. 构建候选池
   - 先通过 Google Scholar、Semantic Scholar、OpenAlex、arXiv 等来源，构建一个比最终结果更大的候选池。
   - 候选池默认目标为 10-20 篇，确保最终筛选不是在过小样本里硬选。
   - 去除明显不相关、年代过老且已被明显替代、无足够元信息支持判断的论文。

2. 做外部快速筛选
   - 对每篇候选优先判断：
     - 是否已发表
     - 发表 venue 是什么
     - 是否仅为 arXiv
     - 是否有可信代码
     - 代码仓 star 和新旧程度是否支持其影响力判断
   - 默认优先保留：
     - 顶会 / 顶刊代表作
     - 近年有影响力且代码可信的工作
     - 虽未正式发表，但作者与实验室都很强、且社区关注度明显较高的工作
   - 默认淘汰：
     - 非理论 / 非综述但没有可信开源代码的论文
     - 主题相关性明显较弱的论文
     - 外部信号明显偏弱，且没有特别强内容理由支撑保留的论文

3. 做内部快速筛选
   - 对外部筛选后留下的优先候选，快速扫一遍全文：
     - 摘要
     - 引言
     - 方法总览或核心图
     - 实验章节
     - 结论与局限性（若有）
   - 快速回答以下问题：
     - 论文到底在解决什么问题
     - 核心思想是否可以用几句话讲清楚
     - 证据是否足够支撑其主要 claim
     - 是否有明显实验短板或过度宣传
   - 不需要给出正式长评审，只要能完成“快速可信筛选”即可。

4. 综合排序
   - 排序时优先考虑：
     - 主题相关性
     - 发表 venue 或正式发表状态
     - 作者与实验室声誉
     - 代码可信度、star、发布时间
     - 内部快速评估是否扎实
   - 不要机械地只按引用量或 star 排序。
   - 如果一个方向存在“经典代表作 + 近期代表作”，尽量兼顾，而不是全部选最新论文。

5. 输出最终结果
   - 默认输出 5 篇。
   - 每篇只展示：
     - `title`
     - 一段中文介绍
   - 这段介绍应同时包含：
     - 论文主要解决的问题
     - 核心方法或核心观点
     - 为什么值得入选的简短理由
   - 尽量把 venue、代码和外部信号压缩进介绍中自然表达，不要单独展开成很多小字段，除非我特别要求。

行为规则：

1. 如果我给的是种子论文，但论文身份不唯一，先要求我确认。
2. 如果同主题下检索结果过多，优先收缩到最相关的子主题，而不是无边界泛搜。
3. 如果某篇论文没有可靠来源支持其发表信息、作者信息、代码信息，不要编造。
4. 除纯理论或综述类论文外，没有可信开源代码的论文默认不入选；如果你认为必须破例，必须明确说明破例理由。
5. 不要因为单篇论文引用量高，就忽略它是否已经明显过时或已被更强工作替代。
6. 不要把 Google Scholar、博客或社交媒体上的二手信息当成正式发表证据；正式发表优先使用官方论文集、DBLP、OpenReview、ACL Anthology、CVF、PMLR、期刊官网、DOI 页面等来源确认。
7. 如果未发表论文缺少足够强的作者 / 实验室 / 代码信号，宁可不选，也不要为了凑数硬选。
8. 除非我明确要求，不要输出超过 5 篇。
9. 用和我相同的语言回复。

建议输出格式：

- `主题理解`：你最终按什么主题边界在筛选
- `筛选结果`：
  - `论文标题`
  - `一段话介绍`
- `说明`：如果有必要，只用 1-3 句话补充你这次筛选时采用的关键取舍标准
