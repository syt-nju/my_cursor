# blog-to-html

把一篇 blog / paper / 长文转成**单文件、自包含、适合阅读**的教学可视化 HTML。

**核心原则：帮助读者 get 到文章的核心思想。** 页面首先要流畅、易读，让读者顺着作者的思路读懂文章；可视化只是让抽象关系更具体，小卡片只是让读者在掌握脉络后自然深入细节。

## 输入输出

- **输入**：URL / markdown 路径 / 粘贴文本（URL 优先用 Read 或 WebFetch 拉全文）
- **输出**：单个 HTML 文件，无外部构建依赖（字体用 CDN，图标用 emoji 或内联 SVG）
- **语言**：面向中文读者，正文 / 标题 / 导航 / 按钮 / 提示文案一律用中文；原文是英文时要翻译，**禁止整页堆英文**。仅在以下情况保留英文原文：专有名词（人名、公司、产品、论文标题）、代码 / 命令 / API、需要原汁原味的金句引用（此时配中文译文或简短解释）
- **源链接**：如果输入提供了原文 URL（或文内可提取到权威源链接），HTML 必须显式引用：
  - 页头或 TL;DR 附近放一个指向原文的「阅读原文 / Source」链接
  - 页脚再出现一次，附作者 / 站点名（若已知）
  - 如果文中引用了其他外链资源，也保留跳转
  - 参考样式：[anthropic-multi-agent-research-system](https://syt-nju.github.io/files/visual-reading/anthropic-multi-agent-research-system/) —— 顶部导航带「原文 ↗」，页脚再列一次「原文：标题 · 站点」

## 流程

### 第 1 步：先确定读者要 get 到什么

动手写 HTML 前，先用 3–5 句话回答：

- 这篇文章最核心的思想是什么？
- 读者最容易卡在哪里：背景、概念、因果链、流程、对比、证据，还是作者的取舍？
- 原文最自然的阅读顺序是什么？哪些内容必须连续读，哪些内容可以做成可展开细节？

页面结构必须服务这个答案。不要为了做可视化而改写文章的重心，也不要把一篇需要连续阅读的文章切成卡片墙。

### 第 2 步：设计一条流畅阅读路径

优先保证读者能顺着页面自然读下去：

```text
先抓住主旨 → 顺着正文理解脉络 → 用图把抽象关系具体化 → 用卡片/表格逐步深入证据和细节
```

具体 section 顺序、数量、命名由原文决定。好的 section 标题应该像文章大纲，不像组件清单。

### 第 3 步：为每块内容选可视化手段

**可视化 pattern 库**（按需取用，不要全用）：

- **静态流程图**：节点 + 连线 + 阶段说明 —— 适合架构 / 时序 / 因果链
- **交互流程图**：节点 + 连线 + 状态切换 —— 仅当交互能解释状态变化时使用
- **可展开卡片网格**：标题秒懂 + 点开看详情 —— 适合并列原则 / 多点经验
- **双栏对比 panel**：左右两列 —— 适合 do/don't、适合/不适合、before/after
- **分层大纲 / mindmap**：树状可折叠 —— 适合层次概念
- **时间线**：横向或纵向 —— 适合发展史、步骤、演进
- **数字 metric card**：大字号 + 单位 + 说明 —— 适合强调关键量化结论
- **引用高亮块**：带装饰的引用 —— 适合保留原作者的金句
- **决策框**：是/否分支或矩阵 —— 适合"该不该用"这类判断题
- **对比表格**：维度 × 方案 —— 适合多方案横评
- **代码 / 公式片段**：带语法高亮 —— 适合技术细节

**选择要点**：

- 先判断这块内容是主线、证据、补充解释，还是可选细节
- 一屏至多一个"视觉锚点"（大动画或大图），其余用轻量手段
- 数字永远值得单独拎出来
- 并列关系用卡片，链式关系用流程图，层级关系用树
- 图必须回答一个具体问题：谁先发生、谁依赖谁、反馈回到哪里、什么时候进入下一步
- 动画必须承载信息变化；如果只是让点沿线移动，删掉，改成静态图和清晰标签
- 主线内容默认展开；可点击卡片用于读者已经 get 到脉络后的逐步深入，不要藏关键论证

### 第 4 步：组织页面（遵循渐进式披露）

三层由浅入深：

```text
第 1 层：TL;DR + 核心思想             ← 30 秒知道文章在讲什么
第 2 层：连续正文 + 少量关键图解       ← 3–5 分钟读懂脉络
第 3 层：卡片、表格、原话、延伸细节    ← 按需深读
```

具体 section 顺序、数量、命名**完全由原文决定**，不要套固定模板。

**长文阅读底线**：

- 连续正文是骨架，图解和卡片是辅助
- 每个复杂图解前必须有自然过渡，说明它为什么出现
- 导航命名应反映读者的理解路径，而不只是组件名
- 如果读者需要按顺序理解，就不要一上来堆术语或架构图；先交代为什么这件事重要

### 第 5 步：视觉与交互规范

**风格底线：克制、低饱和、信息优先。** 不做装饰性炫技，不靠颜色堆砌存在感；强调用字重 / 字号 / 留白，不用颜色。

- **配色**：
  - 主色调参考 Anthropic 风格的柔和浅黄 / 米色系，优先使用暖白、奶油黄、浅砂色、浅赭色；避免纯白大面积刺眼背景
  - 背景、卡片、分割线用同一暖色相的明度层级区分，例如 `#fbf6ea` / `#f4ead7` / `#e8dcc7`，文字用深棕灰或近黑
  - 最多 **1 个低饱和暖色 accent**（用于链接、关键 metric、当前选中态），整页保持同一个色相
  - 语义色（success / warn / danger）按需出现，**同一页面内合计不超过 1 种**，不要又绿又黄又红
  - **禁止**：彩色渐变背景、霓虹 / 荧光色、发光效果（`box-shadow` blur > 20px 的彩色阴影）、彩虹色或多色渐变文字、半透明玻璃拟态堆叠
- **字体**：一套 sans（Inter + Noto Sans SC）+ 一套 mono（JetBrains Mono）；层级靠字重 / 字号 / 行距，避免用颜色做层级
- **交互**：纯 CSS transition + JS 切 class，不引入动画库
- **动效**：仅限「展开 / 折叠 / 淡入 / 短距离位移」，禁止旋转、弹跳缩放、连续 loop 动画、粒子 / 光斑 / 鼠标拖尾
- **动画使用条件**：只有当动画展示了真实状态迁移、队列推进、反馈闭环或时序依赖时才使用；装饰性 packet 动画禁止
- **展开动画**：用 `max-height` / `opacity` / `transform`，不要 `display: none`
- **响应式**：`@media (max-width: 880px)` 转单列

> 自检：把页面截图转成灰度，如果信息层级依然清晰，说明颜色用对了。

### 第 6 步：⚠️ 验证（必做，这步省不掉）

**静态层** —— 写完立刻跑：

```bash
rg '(title|body|hint|example|label|text):\s*"[^"]*"[^"]*"' <file>
rg "(title|body|hint|example|label|text):\s*'[^']*'[^']*'" <file>
```

命中即说明 JS 字符串里有嵌套同类型引号，会让整个 `<script>` parse 失败。
**解法**：中文引用改用 `「」` `『』`，或切换引号类型 / 模板字符串。

其他静态检查：

- 所有 `<script>` / `</script>` 成对
- `getElementById` 对应的 DOM id 都真实存在
- ReadLints 过一遍

**运行时层** —— 绝不能跳过：

- `open <file>` 在浏览器打开
- **F12 → Console 必须无红色报错**（静态检查过了不代表能跑）
- 手动点一遍所有交互元素：展开、播放、重置、锚点跳转
- DevTools 切 mobile 尺寸看响应式

**内容层**：

- 核心数字与原文对齐（别幻觉）
- 所有外链 URL 正确
- TL;DR 真的能让外行 30 秒抓住核心思想
- section 顺序读起来是否顺滑，能不能让读者自然从主旨走到细节
- 最大的一张图是否准确表达原文的真实流程；如果图里节点顺序错了，宁可删图
- 所有交互是否帮助逐步阅读；没有学习价值或打断阅读的交互删掉

### 第 7 步：交付

- 浏览器打开让用户确认
- 列 2–3 个可迭代方向，不主动写 README
- 顺手起一个本地 server 方便阅读，例如在 HTML 所在目录执行 `python3 -m http.server 8000`，把访问地址告诉用户，需要给用户具体ip下的地址而不是localhost

## 论文专项指引

当输入是一篇学术论文时，阅读路径和侧重点与一般博客不同。以下指引**叠加**在上面的通用流程之上，不替代它。

### 叙事主线

```text
这篇论文要解决什么问题 → 已有方案怎么做的、卡在哪 → 本文的做法（核心贡献）→ 有意思的细节 → 实验结论一句话
```

按这条线组织 section，具体展开：

1. **问题定义（轻量）**：用 1–2 段白话说清"为什么需要这篇论文"，不堆术语。如果问题本身抽象，画一张最小示意图让读者秒懂。

2. **已有方案速览（Related Work 可视化）**：
   - 不要照搬 related work 的引用列表，只挑与本文方法形成对比的 2–3 种方案。
   - 每种方案用一张**可展开的卡片**呈现：折叠态显示方案名称 + 一句话定位；展开后包含 **简笔示意图**（流程图 / 框图）+ **2–3 段简短文字**说清"它做了什么、怎么做的、卡在哪"。只有图和一句点评不够——读者需要足够的上下文才能理解这个方案与本文的区别。
   - 目的是让读者在进入本文方法前，已经对"别人怎么做"有画面感，从而理解本文方案的动机。

3. **本文方法（重头戏，花最多篇幅）**：
   - **优先复用原文的图**：论文里通常有精心画好的 architecture / pipeline / 示意图。如果某张图有助于理解，直接引用（`<img>` 加说明），不必自己重画；在图前后用中文补充"这张图在说什么"。
   - **伪代码可视化**：如果论文给了 Algorithm 伪代码，可以用带语法高亮的代码块展示，也可以把关键步骤拆成可交互的分步流程图（点击切换步骤 + 高亮当前行）。
   - **自制图解填补空白**：对于原文只用文字描述、没给图的关键机制，自己画流程图 / 框图辅助理解。
   - 一个复杂方法可以拆成"总览 → 各模块细讲"的渐进结构，但每一步都要回答"为什么这么做"。

4. **有意思的点（可选，方法之后）**：论文中你觉得值得单独拿出来说的内容——巧妙的 trick、反直觉的发现、设计取舍的讨论、与其他工作的有趣联系等——用独立小节呈现。

5. **实验结论（轻量）**：不要逐表逐图搬运数字。用 1–2 句话概括"在哪些 benchmark 上比谁强多少"，或者用一张 metric card 呈现最核心的数字即可。如果论文自己提到了 limitations / failure cases 且说得有道理，也值得一两句话带出来——这往往比数字更能帮读者判断这个方法适不适合自己的场景。读者来这里是快速 view 方法，不是审稿。

### 论文图片处理

- 如果原文 PDF / arXiv 页面上有可直接引用的图片 URL，用 `<img>` 嵌入并标注来源（Figure X from original paper）。
- 如果拿不到图片 URL，用自制 SVG / CSS 图解代替，但要忠于原图的信息。
- 所有图都必须有中文图注，解释这张图在说什么。

## 反面教训

1. **模板优先而非内容优先**：拿着固定模板塞内容，会让不合适的 section 变成累赘、合适的内容没地方放。
2. **中文双引号混 ASCII 双引号**：`"把"重点"加粗"` 让 JS 直接 parse 失败，页面看起来静态。必须 grep 扫一遍。
3. **只做静态检查不开浏览器**：Lint 过不代表 JS 能跑，必须看 Console。
4. **一屏糊太多信息**：违反渐进式披露，第一屏应该 30 秒看懂骨架。
5. **为动画而动画**：如果内容本身是并列关系，硬做时序动画反而混淆结构。
6. **色彩 / 装饰过载**：渐变背景 + 霓虹色 + 多色 accent + 发光阴影叠在一起，读者注意力被视觉抢走，反而看不清内容。色用得越少，信息越突出。
7. **过度规定叙事模板**：每篇文章的阅读路径不同，不要把某一次改稿的结构写成通用结构。
8. **把长文切成卡片墙**：卡片适合承载补充细节，不适合取代连续正文。读者还没 get 到脉络时，不要让他到处点卡片找主线。
9. **画错流程图**：流程图必须忠于原文真实依赖。图如果让文章逻辑更难懂，就删掉或改回文字。
10. **论文当综述写**：目标是帮读者快速 get 方法，不是综述 20 篇 related work。只挑形成对比的少数方案。
11. **搬运实验表格**：大段数字堆砌读者不会看。实验只给最核心结论，一两句话或一张 metric card 就够。
12. **有好图不用**：原文花精力画的 pipeline 图往往比自己重画的更准确，优先复用。

## 心法

> **先问"读者读完应该 get 到什么"，再问"怎样读最顺"，最后才问"用什么视觉呈现能帮他 get 到"。**
>
> 页面结构是答案，不是前提。
