# blog-to-html

把一篇 blog / paper / 长文转成**单文件、自包含、适合阅读**的教学可视化 HTML。

**核心原则：帮助读者 get 到文章的核心思想。** 页面首先要流畅、易读，让读者顺着作者的思路读懂文章；可视化只是让抽象关系更具体，小卡片只是让读者在掌握脉络后自然深入细节。

## 输入输出

- **输入**：URL / markdown 路径 / 粘贴文本（URL 优先用 Read 或 WebFetch 拉全文）
- **输出**：单个 HTML 文件，无外部构建依赖（字体用 CDN，图标用 emoji 或内联 SVG）
- **语言**：面向中文读者，正文 / 标题 / 导航 / 按钮 / 提示文案一律用中文；原文是英文时要翻译，**禁止整页堆英文**。仅在以下情况保留英文原文：专有名词（人名、公司、产品、论文标题）、代码 / 命令 / API、需要原汁原味的金句引用（此时配中文译文或简短解释）
- **源链接**：如果输入提供了原文 URL（或文内可提取到权威源链接），HTML 必须显式引用：
  - 页头或 TL;DR 附近放一个指向原文的「阅读原文 / Source」链接
  - 页脚再出现一次，附作者 / 站点名（若已知）
  - 如果文中引用了其他外链资源，也保留跳转
  - 参考样式：[anthropic-multi-agent-research-system](https://syt-nju.github.io/files/visual-reading/anthropic-multi-agent-research-system/) —— 顶部导航带「原文 ↗」，页脚再列一次「原文：标题 · 站点」

## 流程

### 第 1 步：先确定读者要 get 到什么

动手写 HTML 前，先用 3–5 句话回答：

- 这篇文章最核心的思想是什么？
- 读者最容易卡在哪里：背景、概念、因果链、流程、对比、证据，还是作者的取舍？
- 原文最自然的阅读顺序是什么？哪些内容必须连续读，哪些内容可以做成可展开细节？

页面结构必须服务这个答案。不要为了做可视化而改写文章的重心，也不要把一篇需要连续阅读的文章切成卡片墙。

### 第 2 步：设计一条流畅阅读路径

优先保证读者能顺着页面自然读下去：

```text
先抓住主旨 → 顺着正文理解脉络 → 用图把抽象关系具体化 → 用卡片/表格逐步深入证据和细节
```

具体 section 顺序、数量、命名由原文决定。好的 section 标题应该像文章大纲，不像组件清单。

### 第 3 步：为每块内容选可视化手段

**可视化 pattern 库**（按需取用，不要全用）：

- **静态流程图**：节点 + 连线 + 阶段说明 —— 适合架构 / 时序 / 因果链
- **交互流程图**：节点 + 连线 + 状态切换 —— 仅当交互能解释状态变化时使用
- **可展开卡片网格**：标题秒懂 + 点开看详情 —— 适合并列原则 / 多点经验
- **双栏对比 panel**：左右两列 —— 适合 do/don't、适合/不适合、before/after
- **分层大纲 / mindmap**：树状可折叠 —— 适合层次概念
- **时间线**：横向或纵向 —— 适合发展史、步骤、演进
- **数字 metric card**：大字号 + 单位 + 说明 —— 适合强调关键量化结论
- **引用高亮块**：带装饰的引用 —— 适合保留原作者的金句
- **决策框**：是/否分支或矩阵 —— 适合"该不该用"这类判断题
- **对比表格**：维度 × 方案 —— 适合多方案横评
- **代码 / 公式片段**：带语法高亮 —— 适合技术细节

**选择要点**：

- 先判断这块内容是主线、证据、补充解释，还是可选细节
- 一屏至多一个"视觉锚点"（大动画或大图），其余用轻量手段
- 数字永远值得单独拎出来
- 并列关系用卡片，链式关系用流程图，层级关系用树
- 图必须回答一个具体问题：谁先发生、谁依赖谁、反馈回到哪里、什么时候进入下一步
- 动画必须承载信息变化；如果只是让点沿线移动，删掉，改成静态图和清晰标签
- 主线内容默认展开；可点击卡片用于读者已经 get 到脉络后的逐步深入，不要藏关键论证

### 第 4 步：组织页面（遵循渐进式披露）

三层由浅入深：

```text
第 1 层：TL;DR + 核心思想             ← 30 秒知道文章在讲什么
第 2 层：连续正文 + 少量关键图解       ← 3–5 分钟读懂脉络
第 3 层：卡片、表格、原话、延伸细节    ← 按需深读
```

具体 section 顺序、数量、命名**完全由原文决定**，不要套固定模板。

**长文阅读底线**：

- 连续正文是骨架，图解和卡片是辅助
- 每个复杂图解前必须有自然过渡，说明它为什么出现
- 导航命名应反映读者的理解路径，而不只是组件名
- 如果读者需要按顺序理解，就不要一上来堆术语或架构图；先交代为什么这件事重要

### 第 5 步：视觉与交互规范

**风格底线：克制、低饱和、信息优先。** 不做装饰性炫技，不靠颜色堆砌存在感；强调用字重 / 字号 / 留白，不用颜色。

- **配色**：
  - 主色调参考 Anthropic 风格的柔和浅黄 / 米色系，优先使用暖白、奶油黄、浅砂色、浅赭色；避免纯白大面积刺眼背景
  - 背景、卡片、分割线用同一暖色相的明度层级区分，例如 `#fbf6ea` / `#f4ead7` / `#e8dcc7`，文字用深棕灰或近黑
  - 最多 **1 个低饱和暖色 accent**（用于链接、关键 metric、当前选中态），整页保持同一个色相
  - 语义色（success / warn / danger）按需出现，**同一页面内合计不超过 1 种**，不要又绿又黄又红
  - **禁止**：彩色渐变背景、霓虹 / 荧光色、发光效果（`box-shadow` blur > 20px 的彩色阴影）、彩虹色或多色渐变文字、半透明玻璃拟态堆叠
- **字体**：一套 sans（Inter + Noto Sans SC）+ 一套 mono（JetBrains Mono）；层级靠字重 / 字号 / 行距，避免用颜色做层级
- **交互**：纯 CSS transition + JS 切 class，不引入动画库
- **动效**：仅限「展开 / 折叠 / 淡入 / 短距离位移」，禁止旋转、弹跳缩放、连续 loop 动画、粒子 / 光斑 / 鼠标拖尾
- **动画使用条件**：只有当动画展示了真实状态迁移、队列推进、反馈闭环或时序依赖时才使用；装饰性 packet 动画禁止
- **展开动画**：用 `max-height` / `opacity` / `transform`，不要 `display: none`
- **响应式**：`@media (max-width: 880px)` 转单列

> 自检：把页面截图转成灰度，如果信息层级依然清晰，说明颜色用对了。

### 第 6 步：⚠️ 验证（必做，这步省不掉）

**静态层** —— 写完立刻跑：

```bash
rg '(title|body|hint|example|label|text):\s*"[^"]*"[^"]*"' <file>
rg "(title|body|hint|example|label|text):\s*'[^']*'[^']*'" <file>
```

命中即说明 JS 字符串里有嵌套同类型引号，会让整个 `<script>` parse 失败。
**解法**：中文引用改用 `「」` `『』`，或切换引号类型 / 模板字符串。

其他静态检查：

- 所有 `<script>` / `</script>` 成对
- `getElementById` 对应的 DOM id 都真实存在
- ReadLints 过一遍

**运行时层** —— 绝不能跳过：

- `open <file>` 在浏览器打开
- **F12 → Console 必须无红色报错**（静态检查过了不代表能跑）
- 手动点一遍所有交互元素：展开、播放、重置、锚点跳转
- DevTools 切 mobile 尺寸看响应式

**内容层**：

- 核心数字与原文对齐（别幻觉）
- 所有外链 URL 正确
- TL;DR 真的能让外行 30 秒抓住核心思想
- section 顺序读起来是否顺滑，能不能让读者自然从主旨走到细节
- 最大的一张图是否准确表达原文的真实流程；如果图里节点顺序错了，宁可删图
- 所有交互是否帮助逐步阅读；没有学习价值或打断阅读的交互删掉

### 第 7 步：交付

- 浏览器打开让用户确认
- 列 2–3 个可迭代方向，不主动写 README
- 顺手起一个本地 server 方便阅读，例如在 HTML 所在目录执行 `python3 -m http.server 8000`，把访问地址告诉用户，需要给用户具体ip下的地址而不是localhost

## 论文专项指引

当输入是一篇学术论文时，阅读路径和侧重点与一般博客不同。以下指引**叠加**在上面的通用流程之上，不替代它。

### 论文元信息标注

页面 header 区域必须标注论文的**发表时间**和**发表venue**：

- **发表时间**：至少精确到年月（如 `2026.05`）。
- **发表 venue**：如果论文发表在顶会/顶刊（如 NeurIPS、ICML、ICLR、ACL、EMNLP、AAAI、CVPR、TPAMI、Nature、Science 等），明确标注年份+会议/期刊名（如 `NeurIPS'23`、`ICML'25`）。仅挂 arXiv 未正式发表的标注 `arXiv preprint`。
- **被引论文同理**：在深度卡片或引用论文的折叠态中，标注该论文的发表 venue 和年份（如 `NeurIPS'23`），帮助读者快速判断论文的权威性和时效性。信息来源优先用 deepxiv 的 `venue` 字段或论文自身标注。

### 叙事主线

```text
这篇论文要解决什么问题 → 已有方案怎么做的、卡在哪 → 本文的做法（核心贡献）→ 有意思的细节 → 实验结论一句话
```

按这条线组织 section，具体展开。**跨 section 原则：论文中的具体示例（input-output 样例、case study、running example、反例、failure case 截图等），不管是图还是文本，只要有助于读者理解，都要保留并嵌入对应位置。** 示例是读者从"好像懂了"到"真的懂了"的桥梁，不要省掉。

1. **问题定义（轻量）**：用 1–2 段白话说清"为什么需要这篇论文"，不堆术语。如果问题本身抽象，画一张最小示意图让读者秒懂。如果论文开头就给了 motivating example（一个具体的输入输出或场景），优先用它来让读者"一看就知道在说什么"。

2. **已有方案速览（Related Work 可视化）**：
   - 不要照搬 related work 的引用列表，只挑与本文方法形成对比的 2–3 种方案。
   - 每种方案用一张**可展开的卡片**呈现：折叠态显示方案名称 + 一句话定位；展开后包含 **pipeline / 架构图 + 2–3 段简短文字**说清"它做了什么、怎么做的、卡在哪"。图和文字缺一不可——只有图读者看不懂方案在干嘛，只有文字又缺乏画面感。
   - 图片来源和说明格式遵循下方「论文图片规范」。
   - 目的是让读者在进入本文方法前，已经对"别人怎么做"有画面感，从而理解本文方案的动机。

3. **本文方法（重头戏，花最多篇幅）**：
   - 方法部分的 pipeline / 架构图是全文最重要的图，必须有。图片来源和说明格式遵循下方「论文图片规范」。
   - **伪代码可视化**：如果论文给了 Algorithm 伪代码，可以用带语法高亮的代码块展示，也可以把关键步骤拆成可交互的分步流程图（点击切换步骤 + 高亮当前行）。
   - **具体示例**：如果论文在方法部分给了 running example、输入输出样例、或分步骤的具体实例来演示方法如何工作，必须保留。用高亮块、表格或分步卡片呈现，让读者跟着例子走一遍方法流程。图片型示例（如模型输出截图、可视化结果）直接嵌入并配说明。
   - 一个复杂方法可以拆成"总览 → 各模块细讲"的渐进结构，但每一步都要回答"为什么这么做"。

4. **有意思的点（可选，方法之后）**：论文中你觉得值得单独拿出来说的内容——巧妙的 trick、反直觉的发现、设计取舍的讨论、与其他工作的有趣联系等——用独立小节呈现。

5. **实验结论（轻量）**：不要逐表逐图搬运数字。用 1–2 句话概括"在哪些 benchmark 上比谁强多少"，或者用一张 metric card 呈现最核心的数字即可。如果论文自己提到了 limitations / failure cases 且说得有道理，也值得一两句话带出来——这往往比数字更能帮读者判断这个方法适不适合自己的场景。如果论文给了**定性示例**（如生成结果对比图、case study、可视化对比），挑最能说明方法优劣的 1–2 个嵌入，比干巴巴的数字更直观。读者来这里是快速 view 方法，不是审稿。

### 综述性论文专项（Survey / Review）

当判断输入论文是综述性论文（标题含 survey / review / overview / tutorial，或正文结构以分类梳理多篇工作为主、没有提出自己的新方法）时，采用以下叙事主线和递归策略，**替代上面的常规论文叙事主线**（其余通用流程、图片规范、视觉规范仍然适用）。

#### 综述叙事主线

```text
这个领域要解决什么问题 → 综述的分类框架（taxonomy）→ 按分支递归展开每篇关键论文 → 跨方法对比与趋势 → 开放问题 / 未来方向
```

1. **领域全景（轻量）**：用 2–3 段白话说清"这个领域在做什么、为什么重要"。如果综述给了 taxonomy 图 / 分类树，优先引用原图并配中文说明。没有原图时自制一张分类树或 mindmap，让读者一眼看到全貌。

2. **分类框架（核心骨架）**：
   - 把综述的分类维度提炼成**可折叠的树形导航**或**分栏 tab**，读者能按分支逐个探索。
   - 每个分支 / 类别给一段概述：这类方法的共同思路是什么、核心假设是什么、典型代表是谁。

3. **递归展开关键论文（重头戏）**：
   - 对综述中重点讨论的每篇论文，**按照常规论文叙事主线的深度去生成内容**，具体要求：
     - 问题动机（一两句话）
     - 核心方法的 pipeline / 架构图（优先从原论文取图，遵循「论文图片规范」）
     - 方法讲解：输入什么 → 怎么处理 → 输出什么，让没读过原文的人也能转述
     - 具体示例（如果原论文给了 input-output 样例 / case study，保留）
     - 关键实验结论（一两句话或一张 metric card）
   - 每篇论文呈现为**可展开的深度卡片**：折叠态显示论文标题 + 一句话定位 + 所属分类；展开后是完整的方法解读（图 + 文字 + 示例）。
   - **递归深度控制**：综述中被重点分析的论文（通常占 1 段以上篇幅）做深度展开；仅一两句带过的论文只需一句话定位 + 原文链接，不递归。判断标准：综述作者花了多少篇幅讲它。
   - **论文间关系**：在同一分支内的多篇论文之间，用简短的过渡文字说明演进关系（"A 解决了 X，但遗留 Y → B 针对 Y 做了改进"），让读者看到方法的发展脉络而不是孤立的卡片列表。

4. **跨方法对比**：
   - 用一张**对比表格**（维度 × 方法）汇总各分支 / 各论文的关键差异：核心思路、适用场景、优劣、代表性数字。
   - 如果综述本身给了对比表，优先引用原表并翻译；没有则根据文中信息自制。

5. **趋势与开放问题（轻量）**：用 2–3 段总结综述的结论——领域往哪走、还有什么没解决、作者的观点。

#### 综述的页面组织

```text
TL;DR（这篇综述覆盖什么领域、梳理了多少篇工作、核心结论）
  ↓
分类全景图 / Taxonomy（树或 mindmap，可交互导航）
  ↓
分支 A：概述 → 论文 A1（深度卡片）→ 论文 A2 → …
分支 B：概述 → 论文 B1 → 论文 B2 → …
  ↓
跨方法对比表格
  ↓
趋势 / 开放问题
```

- 顶部放一个**分类导航**（sticky sidebar 或 tab bar），让读者按兴趣跳到对应分支，不必线性通读。
- 每个分支内部仍保持线性阅读顺序（概述 → 各论文 → 小结）。
- 深度卡片默认折叠，读者按需展开——综述论文引用量大，全部展开会淹没读者。

#### 递归取图策略

综述引用的每篇关键论文都需要配图。执行流程：

1. 提取论文的 arXiv ID（或其他可定位的标识）
2. 从 arxiv HTML 版（`https://arxiv.org/html/<id>v<ver>/`）提取图片 URL，用 `curl -sI` 验证 HTTP 200（遵循「论文图片规范 → 图片来源优先级」）
3. 取到图后嵌入对应论文的深度卡片内
4. arxiv HTML 不可用时，检查 GitHub 仓库；所有途径都失败时，根据综述中的文字描述自制 CSS/SVG 示意图

> **工作量提示**：综述论文的递归生成工作量大。如果综述覆盖 >15 篇重点论文，在第 1 步确认读者目标时主动与用户确认：是全部展开，还是只深入用户最关心的几个分支。

#### Sub-Agent 并行调研（涉及多篇论文时必做）

当页面涉及 ≥3 篇需要深度展开的论文时，**必须启动 sub-agent 并行调研每篇论文**，而不是在主流程中逐篇串行获取。这是因为每篇论文的深度调研（获取结构、读取方法 section、提取并验证图片 URL、整理示例）耗时较长，串行执行会严重拖慢整体速度。

**执行方式**：

1. 为每篇需要深度展开的论文启动一个 `generalPurpose` sub-agent（`run_in_background: true`）
2. 每个 sub-agent 的 prompt 必须包含以下调研任务：
   - `deepxiv paper <id> --head` 获取论文结构
   - `deepxiv paper <id> --section <method_section>` 获取方法详情
   - 从 arxiv HTML 版提取所有图片 URL 并逐个 `curl -sI` 验证 HTTP 200
   - 整理返回：核心方法讲解（3-6 段）、已验证图片 URL + 内容描述、具体示例、关键实验数据、局限性
3. 等所有 sub-agent 返回后，用收集到的素材统一构建 HTML
4. 如果某篇论文的 arxiv HTML 图片不可用，sub-agent 应明确说明并建议自制图的内容描述

**每个 sub-agent 必须返回的结构化信息**：

```text
## 论文标题 + venue + 年份
### 核心方法讲解（3-6 段）
- 解决什么问题（动机）
- 核心做法：输入 → 处理 → 输出
- 关键设计选择和为什么
- 与前序工作的区别
### 已验证图片
- [URL] 描述：xxx（HTTP 200 已确认）
- 每张图的详细内容描述（图中每个模块、箭头、数据流）
### 具体示例
- 原文给出的 input-output 样例 / case study
### 关键数据
- 最重要的 2-3 个实验数字
### 局限性
```

> **严禁**：跳过 sub-agent 直接写 HTML 然后每篇论文只给两行介绍。如果你发现自己在为一篇论文写少于 3 段方法讲解且没有配图，停下来——你缺少素材，需要先调研。

### 论文图片规范

**最高优先级：让读者看懂。** 图片存在的唯一理由是帮读者理解方法，所有规则都服务这个目标。

#### 引用论文必须配图 + 方法讲解

**每一篇被提到的论文 / 系统都必须有原论文的图 + 清晰的方法讲解。** 不能只写一句「XX 系统做了 YY」就带过。至少要讲清楚 method：

- **它要解决什么问题**（动机，一两句话）
- **它的核心做法是什么**（method：输入什么 → 怎么处理 → 输出什么）
- **与本文主线的关系**（它在当前叙事中扮演什么角色）

如果读者没读过这篇被引论文，看完你的图 + 说明后应该能向别人转述它的核心方法。做不到就说明不够。

#### 图片来源优先级

1. **原论文的图优先**。作者精心画的 pipeline / 架构图准确性和信息密度远超自制图，且可信度更高。动手画任何图之前，先尝试拿原图。
2. **取图方法（arXiv 论文）**：
   - **首选 arxiv HTML 版**：图片 URL 格式为 `https://arxiv.org/html/<paper_id>v<version>/<filename>.png`。获取方式：用 `curl` 或 `WebFetch` 拉 `https://arxiv.org/html/<paper_id>v<version>/` 的 HTML 源码，从 `<img src="...">` 标签提取相对路径（通常形如 `x1.png`、`figures/xxx.png` 等），拼接为完整 URL。
   - **验证图片可用**：拼接后必须用 `curl -sI <url>` 检查返回 `HTTP 200` 且 `content-type` 为 image，确认能加载再嵌入 HTML。
   - **⚠️ 不要用 ar5iv**：`ar5iv.labs.arxiv.org` 是第三方 LaTeX→HTML 渲染服务，**不对外暴露图片静态文件**——请求其图片路径会 redirect 到论文摘要页，导致 `<img>` 全部 404。
   - **回退路径**：arxiv HTML 版本不可用时（部分旧论文未生成 HTML），检查论文的 GitHub 仓库 `img/` 或 `figures/` 目录。
   - Related work 引用的其他论文同理——去对应论文的 arxiv HTML 页面拿图。
3. **自制图解填补空白**：仅当原文确实没有对应的图、且所有取图途径都不可用时，才用 CSS / SVG 自制示意图，且必须忠于原文描述的信息。

#### Pipeline / 架构图

Related work 和本文方法部分，**每个方案都必须配 pipeline 或架构图**——这类图是读者理解方法流程最直接的方式。优先从原论文获取；找不到原图时自制 CSS / SVG 图；但如果方案本身不以 pipeline 为核心（例如纯数学推导），不必强凑。

#### 图片说明格式

**每张嵌入的图都必须配可展开的中文说明卡片**（无例外）。图片本身信息密度高但读者未必看得懂——说明卡片是从「看到图」到「理解图」的桥梁。

实现方式：图片下方放一个可点击展开的说明区域（`展开说明 ▸` / `收起 ▴`），采用**渐进式结构**：

- **折叠态（默认可见）**：紧跟图片的一行 caption，一句话概括这张图整体在讲什么——读者扫一眼就知道图的主题。
- **展开态（点击深入）**：
  - 按图中的**关键模块 / 阶段 / 节点逐个分点说明**，每个点包含：这个模块是什么、做了什么、为什么这么设计、与其他模块的数据流关系。
  - 如果图中有箭头/连线，说明每条连线代表什么数据或控制流。
  - 如果图中有颜色/分区，说明每个区域的含义。
  - **最低深度**：展开态至少 3–5 个分点（简单图除外），每点 1–2 句话。复杂架构图应逐节点/逐阶段展开。
  - 结尾可加一句「一句话总结」收束。

> 判断标准：一个没读过原文的同学，看完图和展开说明后，能向别人转述这个方法的完整流程吗？如果不能，说明不够。

> 实现提示：用简单的 `<span class="fig-toggle" onclick="toggleFigDetail(this)">展开说明 ▸</span>` + `<div class="figure-detail">` 容器即可，通过 JS 切换 `.active` 类控制 `max-height` 展开。

## 反面教训

1. **模板优先而非内容优先**：拿着固定模板塞内容，会让不合适的 section 变成累赘、合适的内容没地方放。
2. **中文双引号混 ASCII 双引号**：`"把"重点"加粗"` 让 JS 直接 parse 失败，页面看起来静态。必须 grep 扫一遍。
3. **只做静态检查不开浏览器**：Lint 过不代表 JS 能跑，必须看 Console。
4. **一屏糊太多信息**：违反渐进式披露，第一屏应该 30 秒看懂骨架。
5. **为动画而动画**：如果内容本身是并列关系，硬做时序动画反而混淆结构。
6. **色彩 / 装饰过载**：渐变背景 + 霓虹色 + 多色 accent + 发光阴影叠在一起，读者注意力被视觉抢走，反而看不清内容。色用得越少，信息越突出。
7. **过度规定叙事模板**：每篇文章的阅读路径不同，不要把某一次改稿的结构写成通用结构。
8. **把长文切成卡片墙**：卡片适合承载补充细节，不适合取代连续正文。读者还没 get 到脉络时，不要让他到处点卡片找主线。
9. **画错流程图**：流程图必须忠于原文真实依赖。图如果让文章逻辑更难懂，就删掉或改回文字。
10. **论文当综述写**：目标是帮读者快速 get 方法，不是综述 20 篇 related work。只挑形成对比的少数方案。
11. **搬运实验表格**：大段数字堆砌读者不会看。实验只给最核心结论，一两句话或一张 metric card 就够。
12. **有好图不用**：原文的 pipeline 图比自己重画的更准确、更可信，务必优先复用（见「论文图片规范」）。
13. **图片说明敷衍**：只写一句「如图所示」等于没写。读者需要知道图里每个关键模块在干嘛，说明不够就等于图白贴了。
14. **引用论文只给一句话**：提到一篇论文 / 系统却不配图、不讲 method，读者无法理解它在干什么，等于白提。每篇被引论文至少要有原图 + 方法讲解，让没读过原文的人也能看懂。
15. **丢掉论文里的具体示例**：论文给的 input-output 样例、case study、running example 是帮读者"真的理解"方法的最佳素材。如果原文有且有助于理解，不要省掉——抽象描述 + 具体示例才完整。
16. **综述当论文列表写**：综述的价值是梳理关系和脉络，不是堆引用。每篇被重点提及的论文都要递归做深度展开（图 + 方法讲解 + 示例），只给一句话摘要等于浪费综述的结构。
17. **综述不分类就铺开**：没有 taxonomy / 分支导航就直接列论文，读者会迷失在几十篇工作里。先给全景分类，再按分支展开。
18. **综述全部展开**：引用 30 篇论文全部深度展开会淹没读者。默认折叠，让读者按需探索；超过 15 篇重点论文时先与用户确认范围。
19. **用 ar5iv 图片 URL**：ar5iv 不对外暴露静态图片文件，请求其路径会 redirect 到论文摘要页，导致所有 `<img>` 显示空白。必须用 `https://arxiv.org/html/<id>v<ver>/<file>.png` 并用 `curl -sI` 验证 200 后才嵌入。
20. **可展开卡片的 onclick 绑在容器上**：如果把 `onclick="this.classList.toggle('expanded')"` 绑在整个卡片容器（如 `.deep-card`）上，点击卡片内部的任何子元素（按钮、文字、图片）都会冒泡到容器触发收起。**正确做法**：onclick 只绑在卡片头部（`.header`），或用 JS 事件监听 + `e.stopPropagation()` 阻止内部点击冒泡。嵌套的可展开元素（如图片说明按钮）必须 `stopPropagation`。

## 心法

> **先问"读者读完应该 get 到什么"，再问"怎样读最顺"，最后才问"用什么视觉呈现能帮他 get 到"。**
>
> 页面结构是答案，不是前提。
