# Download Paper

帮我完成一篇论文的检索、归类、下载与会议信息研判。每次调用只处理一篇论文。

主要目标：

- 确认论文的准确标题、arXiv 链接、PDF 链接与基础元信息。
- 判断这篇论文是否属于 `papers/` 下已有的某个主题目录，并把 PDF 下载到正确位置。
- 如果存在相关开源代码，下载到 `code/` 下。
- 根据 arXiv 时间、代码仓活跃情况与会议信息，评估论文的大致发表时间。
- 搜索 OpenReview 与相关顶会/顶刊，判断论文是否已中会、已发表，或仍处于预印本/审稿阶段。
- 默认并行启动 subagent 缩短检索时间，尤其是“是否已发表 / 是否顶刊顶会”的判断。

执行策略：

1. 先轻量确认论文身份，再并行检索
   - 如果我提供的是标题、关键词、arXiv id、arXiv URL、PDF URL 中的任意一种，先统一解析成唯一的一篇论文。
   - 如果候选结果不唯一，先给我最多 3 个候选并要求我确认，不要直接猜。
   - 在论文身份不唯一前，不要下载 PDF、不要 clone 代码、不要创建新目录。
   - 如果我已经给了明确的 arXiv id、arXiv URL 或 PDF URL，不要再做不必要的大范围搜索。

2. 论文身份唯一后，默认并行启动最多 4 个 subagent
   - `metadata-agent`：负责查 arXiv、官方 PDF、作者、提交时间、版本时间、DOI 等基础元信息。
   - `topic-agent`：负责检查 `papers/` 现有目录、论文清单与说明文档，判断最合适的主题目录。
   - `code-agent`：负责查找官方或高可信开源代码、项目主页、作者主页与仓库线索。
   - `venue-agent`：负责查是否已被会议/期刊接收或发表，并给出“事实 vs 推断”的结论。
   - 能并行就并行，不要把这些网页检索串行慢慢做。

3. 主 agent 负责汇总与执行
   - 汇总各 subagent 的结果，只在确实需要我做决定时与我交互。
   - 在信息足够明确后，直接执行下载与目录创建，不要只停留在建议层面。
   - 如果多个 subagent 结论冲突，优先采用官方来源或可核验来源。

4. subagent 返回内容要求
   - 只返回和主任务直接相关的结论，不要长篇背景介绍。
   - 明确区分：
     - 已确认事实
     - 候选结论
     - 未确认项
   - 每个关键结论都尽量附来源链接或来源名称。
   - 如果没找到可靠证据，要尽快明确写“未找到可靠证据”，不要无限扩展搜索。

工作流：

1. 识别论文
   - 优先使用 arXiv 的官方摘要页与 PDF 页，记录：
     - 标题
     - 作者
     - arXiv id
     - arXiv 摘要页
     - PDF 链接
     - 首次提交时间
     - 最近更新时间
     - DOI（如果有）
   - 如果找不到 arXiv 版本，要明确告诉我，并询问我是否改为使用会议或期刊官网 PDF 继续。

2. 主题归类
   - 检查 `papers/` 下的一级主题目录，并结合目录名、已有论文文件名、主题清单或说明性 markdown 文件来理解每个主题的含义。
   - 判断当前论文是否明显属于某个已有主题。
   - 如果判断比较确定，直接使用该主题目录。
   - 如果不确定或现有主题都不合适，不要擅自新建目录；先向我展示：
     - 最多 3 个最接近的已有主题及理由
     - 一个 `新建主题` 选项
   - 如果我选择新建主题，再询问我最终的文件夹名，并在 `papers/` 下创建它。

3. 下载 PDF
   - 将论文 PDF 下载到选定的 `papers/<topic>/` 目录。
   - 有 arXiv id 时，优先使用文件名格式：`<arxiv_id>_<sanitized_title>.pdf`。
   - 没有 arXiv id 时，使用简洁稳定的英文文件名，不要使用过长或包含特殊字符的名字。
   - 如果目标文件已存在，先告诉我并询问要跳过、覆盖，还是保留两份。
   - 下载完成后确认文件已保存成功，并汇报最终路径。

4. 下载代码
   - 搜索论文相关的开源代码，优先级从高到低：
     - arXiv 页面、项目主页、作者主页、OpenReview 页面中明确给出的官方链接
     - Papers with Code
     - 作者或机构名下的 GitHub 仓库
   - 如果存在多个可能仓库，但无法确认哪个是官方实现，先列出候选并让我确认，不要直接 clone 不确定的仓库。
   - 如果确认存在官方或高可信度仓库，将其下载到 `code/<repo_name>/`。
   - 如果本地已存在同名仓库，先汇报现有路径，不要重复下载，除非我要求更新或重下。
   - 如果没有找到可信代码仓，明确说明“未找到可信开源代码”。

5. 发表时间研判
   - 综合以下信息给出论文的大致发表时间与阶段判断：
     - arXiv 首次提交时间
     - arXiv 后续版本时间
     - 代码仓创建时间、最近活跃时间、release/tag 信息
     - OpenReview、会议官网、期刊官网、论文集页面的投稿/接收/发表时间
   - 区分“事实”和“推断”，不要把推断写成已确认事实。
   - 给出一个简洁结论，例如：
     - `预计为 2025 Q4 投稿 / 2026 顶会结果`
     - `已在 ICML 2026 接收`
     - `已发表于 TMLR`
     - `目前更像是未中会的 arXiv 预印本`
   - 给出你的置信度：高 / 中 / 低。

6. 会议信息与期刊信息检索、快捷判定
   - 不要一开始就全网泛搜，优先走下面这条快查路径：
     - 精确标题匹配 `OpenReview / DBLP / Crossref / OpenAlex`
     - 如果查到 DOI，继续用 DOI 跳转出版社或期刊官方页面
     - 如果是会议论文，优先查对应官方论文集或官方收录页
     - 如果是期刊论文，优先查期刊官网、出版社页面或 DOI 落地页
   - 如果快查没有命中，再搜索与论文方向最相关的主流顶会/顶刊官方页面、论文集或检索站点，例如：
     - ICLR / ICML / NeurIPS / AAAI / KDD
     - ACL / EMNLP / NAACL / COLM / TACL
     - CVPR / ICCV / ECCV / TPAMI / IJCV
     - TMLR / JMLR / JAIR / MLJ / Nature Machine Intelligence
   - 会议方向的优先级：
     - `OpenReview`：ICLR、TMLR、COLM 等
     - `PMLR`：ICML、AISTATS、COLT 等
     - `NeurIPS Proceedings`
     - `ACL Anthology`
     - `CVF Open Access`
     - `AAAI / ACM DL / IEEE Xplore / KDD` 官方页面
   - 期刊方向的优先级：
     - `DOI -> 期刊官网 / 出版社页面`
     - `DBLP / OpenAlex / Crossref` 的精确标题结果
     - 作者主页或机构主页上的正式 publication 列表
   - 优先使用官方来源、论文集页面、OpenReview、ACL Anthology、CVF、PMLR、DBLP、Crossref、OpenAlex、作者主页或机构页面作为证据。
   - 如果在官方页、DOI 落地页、OpenReview 接收页、官方论文集页面中任意一个命中，并且标题/作者匹配，可视为“已确认”。
   - 如果只在 Google Scholar、Semantic Scholar、博客、社交媒体或非官方转载中出现，不算确认。
   - 如果已确认是期刊论文，再额外判断该期刊是否属于该子领域常见顶刊；若存在学科口径差异，要明确写“是否算顶刊依子领域标准而定”。
   - 如果只能找到传闻、非官方转载或二手讨论，不能当成确认中会或确认发表。
   - 如果没有找到可靠证据，要明确写“未确认中会/未确认发表”，并说明已检查过哪些来源。

行为规则：

1. 每次调用只处理一篇论文。
2. 只在以下情况与我交互：
   - 论文身份不唯一
   - 主题归类不确定
   - 需要新建主题
   - 代码仓不唯一或不确定
   - 本地已存在同名文件或仓库
3. 在信息足够明确后，直接执行下载与目录创建，不要只停留在建议层面。
4. 不要编造 arXiv 链接、代码仓、会议结果、期刊结果或发表时间。
5. 结论里要明确哪些是已确认信息，哪些是基于证据的推断。
6. 优先复用现有目录结构与命名风格，不要随意改变 `papers/` 和 `code/` 的组织方式。
7. 用和我相同的语言回复。

输出格式：

- `论文确认`：标题、arXiv、主题目录
- `已执行操作`：PDF 路径、代码路径、是否有跳过项
- `时间研判`：关键事实、推断结论、置信度
- `会议信息`：已确认结果或候选会议/期刊与证据
- `待确认事项`：仅在确实需要我做决定时列出
