# Extract Function

## Overview

将代码框架中某些具有独立逻辑的子功能抽取出来，形成独立的模块或服务文档。提取后的目标文档格式需参考 `service_doc/polaris.md` 的结构。

该指令设计为**两步执行（Two-Phase）**的工作流：
- **第一阶段（Phase 1）**：分析代码，提取功能，生成初始文档，并构建自动化测试样例。
- **第二阶段（Phase 2）**：在用户运行测试样例后，读取测试结果，对初始文档和示例代码进行 Review 和 Refine。

---

## Phase 1: 首次执行（提取与测试构建）

请按照以下流程执行功能的初步抽取：

1. **明确功能**
    - 分析原代码框架，明确要抽取的功能模块的核心职责、输入参数及输出结果。
    - 为抽取出的功能撰写简明扼要的说明。
    - 当一个功能需要另外一个功能的相关知识时，允许通过引用另一个功能文档将其外包出去。

2. **确定环境依赖**
    - 梳理该功能所依赖的外部第三方库、系统环境或内部配置文件。
    - 列出明确的安装命令或配置步骤。

3. **撰写示例代码**
    - 提取并重构出核心代码，要求简洁明了，又保证功能的完整性和独立性。

4. **创建测试文件**
    - 针对示例代码构建独立的测试样例脚本。
    - 测试样例的终端返回结果应被自动保存至本地文件（例如：重定向输出至日志文件或在代码中写入 `.log`/`.json` 文件）。
    - 提示用户去运行该测试文件，并告知运行完成后再次调用本指令进行第二阶段。

---

## Phase 2: 第二次执行（测试结果 Review 与文档 Refine）

在用户运行完测试样例后执行：

1. **读取测试结果**
    - 读取第一阶段生成的测试结果文件（如日志文件等）。
    - 分析代码的实际返回情况、是否出现异常以及输出格式。

2. **代码与文档 Refine**
    - 根据真实的返回结果，补充或修改文档中关于“返回基本信息”的描述。
    - 如果测试发现了潜在 bug 或不够鲁棒的地方，对示例代码进行相应的修正和完善。
    - 确保文档结构最终完全符合目标格式。

---

## Target Document Format

最终生成的文档应严格参考 `service_doc/polaris.md`：

```markdown
# [功能名称]

[简短的功能描述]

## 📦 安装依赖

```bash
[依赖安装命令]
```

## 💻 示例代码

```python
[包含完整功能且符合规范的示例代码]
```

## 📊 返回结果说明 (可选，根据测试结果 Refine)

[函数返回基本信息的详细补充说明]
```

---

## Review Checklist

**Phase 1 Checklist:**
- [ ] 抽取的功能是否完全独立，描述是否清晰？
- [ ] 环境依赖是否收集完整？
- [ ] 是否已生成测试样例文件，且能够将结果自动保存到本地文件？
- [ ] 是否已提示用户执行测试文件并进行 Phase 2？

**Phase 2 Checklist:**
- [ ] 是否已正确读取并分析测试结果？
- [ ] 是否根据实际运行结果对示例代码进行了 Refine？
- [ ] 文档中是否已准确补充了函数的“返回基本信息”？
- [ ] 最终文档格式是否严格遵循了目标规范？
